The Map and The Locks

Understand the tree. Control who has the keys. | 👤 By JR Nation

Windows treats its core files like dark magic, hiding them behind scary warnings and arbitrary drive letters. Linux lays the entire engine bare in a logical, open structure called the Filesystem Hierarchy Standard (FHS). To truly master the terminal, you need to know what these folders do and, more importantly, how Linux decides who is allowed to touch them.

1. The Linux Directory Tree

Everything in Linux starts from the Root, represented by a single forward slash: /. From this root, the entire system branches out.

Directory What lives inside?
/bin & /sbin Binaries. The actual executable programs and core system commands (like ls and cd).
/etc Configuration Files. The brain of the system. Network settings, passwords, and bootloader text files live here.
/home Your Sanctuary. Your documents, downloads, games, and personal dotfiles. This is where you have total power.
/var Variable Data. Files that grow and change continuously, like system logs and crash reports.
/usr User Programs. The massive folder where most of your installed applications and shared icons are stored.
/opt Optional Software. Third-party proprietary apps (like Chrome or Discord) often install themselves here.
/dev Device Files. The interface to your hardware. Your hard drive, keyboard, and even the mouse cursor are represented as files here.
/proc Process Information. A virtual filesystem that provides real-time information about running processes and system resources.
/tmp Temporary Files. A chaotic junk drawer where programs can store temporary data. It gets wiped on reboot.
/boot Boot Files. The critical files that start your operating system, including the Linux kernel and initial RAM disk.
/lib Shared Libraries. The essential code that programs need to run. If a program is a car, these are the parts it needs to function.
/mnt & /media Mount Points. Temporary locations where external drives, USB sticks, and network shares are accessed.
/srv Service Data. Data for services like web servers and databases. If you run a local server, this is where its files will live.
/sys System Information. Another virtual filesystem that provides information about the system's hardware and kernel interfaces.
/lost+found Lost and Found. A safety net for recovering corrupted files after a crash. It’s usually empty unless something has gone wrong.
/root The Superuser's Home. The personal directory for the root user (the all-powerful administrator). It is kept separate from /home for maximum security.

2. Reading Permissions (rwx)

If you run the command ls -la in your terminal, you will see a confusing string of letters next to every file, looking something like this: -rw-r--r--. This is the Linux security vault.

Permissions are broken down into three actions:

  • r (Read): Can you open the file and look at it?
  • w (Write): Can you edit or delete the file?
  • x (Execute): Can you run this file as a program or script?

These permissions are handed out to three different groups in this exact order: the User (the owner), the Group (other associated accounts), and Others (everyone else/guests).

💡 Practice Changing Locks: Want to see how file permissions actually work? Open the JR Nation Terminal Playground and practice creating files with touch, and then modifying their permissions with chmod.

3. Changing the Locks (chmod)

If you pull a random script off GitHub, Linux will aggressively block it from running by default to protect you from yourself. You have to explicitly hand over the keys and give it permission to execute. You do this using the chmod (Change Mode) command.

$ chmod +x install_script.sh

# Adds the (+x) "Execute" permission, allowing the file to run as a program.

⚠️ The 777 Danger: You might see tutorials telling you to run chmod 777 filename. In binary math, 7 means "give Read, Write, and Execute power." Running 777 means giving that absolute power to the User, the Group, AND to Others. It is a terrible security practice. Avoid doing this unless absolutely necessary for troubleshooting.

4. Claiming Ownership (chown)

Sometimes you might move a file using sudo, and suddenly your normal user account is locked out of it because the file is now owned by the all-powerful "root" administrator. To steal ownership back, use chown (Change Owner).

$ sudo chown jaya:jaya my_locked_file.txt

# Gives ownership back to the user 'jaya' and the group 'jaya'.
