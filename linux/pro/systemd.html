Systemd: Master the Daemons

Control the invisible puppet master running your operating system. | 👤 By JR Nation

When you turn on your computer, a massive chain reaction happens. The kernel wakes up the hardware, but then it hands control over to the Init System. On almost all modern Linux distributions, this Init System is called Systemd.

Systemd is the puppet master. It is Process ID 1 (PID 1). It is responsible for starting your network connection, launching your login screen, managing your Bluetooth, and keeping your web servers running in the background. If you want to automate your machine, you must learn how to speak to it.

1. What is a Daemon?

In Linux, a background program that runs without a graphical interface and waits silently for something to happen is called a Daemon. (Usually pronounced "Demon").

You can usually spot them because their names end with the letter 'd'. For example:

  • sshd: The Secure Shell Daemon (waiting for remote logins).
  • httpd: The HTTP Daemon (waiting to serve web pages).
  • bluetoothd: The Bluetooth Daemon (waiting to connect to your headphones).

2. The Master Command: systemctl

To control these Daemons, you use the systemctl command. This is how you check if a service is running, crashed, or completely dead.

$ systemctl status bluetooth

# Shows you exactly what the Bluetooth daemon is doing right now, complete with error logs if it crashed.


$ sudo systemctl restart nginx

# Kills the Nginx web server and immediately boots it back up. Crucial for applying new config files.

3. The Trap: Start vs. Enable

This is the number one mistake beginners make when trying to set up a server or a background task. There is a massive difference between "starting" a service and "enabling" it.

  • sudo systemctl start [service]: Turns the daemon on right now. But as soon as you reboot your computer, it will be dead again.
  • sudo systemctl enable [service]: Tells Systemd to automatically turn this daemon on every single time the computer boots up. (Note: this does not start it right now).
💡 The Pro Move: You can do both at the exact same time by adding the --now flag:

sudo systemctl enable --now bluetooth

4. Forging Your Own Daemon

Remember that Auto-Updater bash script we wrote in the Terminal section? What if you want it to run automatically in the background, without you ever having to manually type the execution command?

You can write your own custom Systemd service file. Let's create a daemon that runs your script. (Note: If you are using Arch Linux, remember our previous warning about auto-updates breaking the rolling release cycle!)

  1. Create a new file in the systemd directory: sudo nano /etc/systemd/system/autoupdate.service
  2. Paste in this blueprint:

[Unit]

Description=My Custom Auto-Updater Daemon

After=network.target

# 'After' tells Systemd to wait until the internet connects before trying to run this.


[Service]

ExecStart=/home/jrnation/update_my_pc.sh

Restart=on-failure

# 'ExecStart' is the absolute path to the script you want to run.


[Install]

WantedBy=multi-user.target

Save the file. Then, tell Systemd to reload its brain so it notices your new file, and enable your custom daemon:

$ sudo systemctl daemon-reload

$ sudo systemctl enable --now autoupdate

Congratulations. You have moved beyond just using the OS. You are now architecting it.
