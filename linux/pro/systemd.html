Systemd: Master the Daemons

Control the invisible puppet master running your operating system. | 👤 By JR Nation

When you turn on your computer, a massive chain reaction happens. The bootloader fires up the kernel, the kernel wakes up the hardware, but then the kernel immediately hands absolute control over to the Init System. For decades, Linux used a slow, sequential system of shell scripts called SysVinit. But as computers got faster and parallel processing became the standard, the Linux ecosystem needed a modern engine.

Enter Systemd. Introduced by Lennart Poettering in 2010, it sparked the biggest civil war in Linux history. Purists hated it because it was massive and complex, but enterprise engineers loved it because it was blisteringly fast. Today, Systemd has won the war. It is the default init system on almost every major distribution.

Systemd is the ultimate puppet master. It is Process ID 1 (PID 1). It is responsible for starting your network connection, launching your login screen, managing your Bluetooth, and keeping your web servers running in the background. If you want to automate your machine, you must learn how to speak its language.

1. What is a Daemon?

In the Unix world, a background program that runs without a graphical interface and waits silently in the shadows for something to happen is called a Daemon. (Usually pronounced "Demon").

They are the invisible gears turning the machine. You can usually spot them because their process names end with the letter 'd'. For example:

  • sshd: The Secure Shell Daemon (waiting silently for remote network logins).
  • httpd: The HTTP Daemon (waiting to serve web pages to incoming browsers).
  • bluetoothd: The Bluetooth Daemon (waiting to connect to your wireless headphones).

2. The Master Command: systemctl

To control these Daemons, you use the systemctl command. This is how you check if a service is running flawlessly, if it crashed, or if it is completely dead.

$ systemctl status bluetooth

# Shows you exactly what the Bluetooth daemon is doing right now, complete with its memory footprint and recent error logs.


$ sudo systemctl restart nginx

# Kills the Nginx web server and immediately boots it back up. Crucial for applying new config files.

3. The Trap: Start vs. Enable

This is the number one mistake beginners make when trying to set up a server or a background task. There is a massive, fundamental difference between "starting" a service and "enabling" it.

  • sudo systemctl start [service]: Turns the daemon on right now. But as soon as you reboot your computer, it will be dead again.
  • sudo systemctl enable [service]: Tells Systemd to create a symlink to automatically turn this daemon on every single time the computer boots up. (Note: this does not start it right now).
💡 The Pro Move: You can do both at the exact same time by adding the --now flag:

sudo systemctl enable --now bluetooth

4. Reading the Matrix: Journalctl logs

When a background daemon crashes, it doesn't pop up a graphical error message on your screen. It dies in the dark. Systemd captures the dying words of every daemon and writes them to a highly compressed, centralized binary log called the Journal.

To read these logs and figure out why your server crashed, you use the journalctl command.

$ journalctl -u nginx.service --since "1 hour ago"

# Filters the massive log matrix to only show messages from the Nginx web server over the last hour.

5. Forging Your Own Daemon

Remember that Auto-Updater bash script we wrote in the Terminal section? What if you want it to run automatically in the background, without you ever having to manually type the execution command?

You can write your own custom Systemd service file. Let's create a daemon that runs your script like a native Linux process.

  1. Create a new file in the systemd directory: sudo nano /etc/systemd/system/autoupdate.service
  2. Paste in this architectural blueprint:

[Unit]

Description=My Custom Auto-Updater Daemon

After=network.target

# 'After' tells Systemd to wait until the internet connects before trying to run this.


[Service]

ExecStart=/home/jrnation/update_my_pc.sh

Restart=on-failure

# 'ExecStart' is the absolute path to the script you want to execute.


[Install]

WantedBy=multi-user.target

# 'multi-user.target' tells systemd to load this when the system hits the standard runlevel (normal multi-user operation).

Save the file. Then, tell Systemd to reload its brain so it parses your new text file, and finally enable your custom daemon:

$ sudo systemctl daemon-reload

$ sudo systemctl enable --now autoupdate

Congratulations. You have moved beyond just using the OS. You are now architecting it.
