User Management: The VIP List

You built the walls. Now you need to decide who gets the keys. | 👤 By JR Nation

A firewall keeps the attackers out of the fortress. But what about the people (or programs) already inside? If you let a friend use your computer, or if you host a Minecraft server for strangers, you don't want them having the power to delete your operating system.

Linux is inherently a multi-user operating system. It was built from the ground up to allow hundreds of people to connect to the same machine simultaneously without ever seeing each other's files. Mastering the Principle of Least Privilege—giving a user exactly enough power to do their job and absolutely nothing more—is the hallmark of a true System Administrator.

1. Spawning New Users (useradd)

Creating a new user account via the terminal requires you to tell the system exactly how to set them up. The universal command for this is useradd.

$ sudo useradd -m -s /bin/bash guest_user

# -m tells Linux to physically create a /home/guest_user directory for them.

# -s sets their default terminal shell to Bash.

This user can now log in, download files to their own /home folder, and run basic apps. However, if they type sudo, the system will reject them and log the incident. They are completely contained.

2. Changing Passwords & Usernames

When you first create a user, their account is locked until they have a password. Or, maybe an existing user forgot theirs and needs you to reset it.

$ passwd

# Changes YOUR current password. It will ask for your old one first.


$ sudo passwd guest_user

# As the admin, you can forcefully overwrite anyone else's password.

What if they want to change their actual username? You use the usermod command, but you also have to rename their /home folder to match the new name, otherwise things get messy.

$ sudo usermod -l new_name -d /home/new_name -m old_name

# -l renames the account. -d and -m move all their old files to the new home folder.

3. The Group System

Managing permissions for 50 individual users is exhausting. Instead, Linux uses Groups. You apply permissions to a Group (like "developers" or "audio"), and then simply drop users into those groups.

Some important default groups include:

  • wheel or sudo: The gods. Anyone in this group can execute root commands.
  • docker: Allows the user to run containerized applications without needing sudo.
  • www-data: The standard group used by web servers like Nginx or Apache.

4. The usermod Trap (-aG)

When you want to grant a user new powers by adding them to a group, you use the usermod (User Modify) command. But there is a massive trap here that catches almost every beginner.

⚠️ The Missing "a" Disaster:
If you type sudo usermod -G sudo jaya, Linux will add you to the sudo group, and immediately delete you from every other group you were in. This will break your audio, your video drivers, and your network access.

You MUST include the -a (Append) flag.

The correct way to add a user to a group without destroying their other permissions is:

$ sudo usermod -aG sudo guest_user

# -a (Append) -G (Groups). Safely adds them to the VIP list.

5. Evicting Users (userdel)

When someone's time in your fortress is up, you need to revoke their access. You can delete their account using the userdel command. However, if you just run sudo userdel guest_user, the system deletes their login ability, but leaves all their files sitting on your hard drive.

To completely wipe their existence from the server, you need to use the recursive flag:

$ sudo userdel -r guest_user

# The -r flag deletes the user AND wipes their entire /home directory off the disk.

6. Pro-Tier: Service Accounts

Users aren't always humans. If you install a database like MySQL or a web server, Linux actually creates a new "user account" specifically for that software.

Why? Security isolation. If a hacker finds a vulnerability in your web server and breaks in, they don't break in as *you*. They break in as the www-data user account. Because the www-data user is isolated and has zero sudo privileges, the hacker is trapped inside the web server folder and cannot touch your personal files or the core system.
