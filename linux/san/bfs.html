Basic Firewall: Closing the Gates

Your system has 65,535 doors. Stop leaving them wide open. | 👤 By JR Nation

Imagine your computer is a massive fortress. In the world of networking, this fortress has exactly 65,535 doors, officially known as Ports. Every time you connect to the internet, applications use these specific doors to send and receive packets of data. Your web browser exclusively uses Port 80 for HTTP and Port 443 for HTTPS. Your email client uses Port 25. Multiplayer games and background services dynamically claim thousands of other random doors.

A Firewall is the digital bouncer standing outside your fortress. Its entire job is to inspect every single packet of data trying to enter or leave, check your established rulebook, and decide whether to let it through or violently drop it into the void. On a fresh Linux installation, the bouncer is usually asleep, meaning all doors are theoretically accessible if a service is listening behind them. It is time to wake him up and hand him the VIP list.

1. The Lore: Iptables vs. UFW

To understand what we are doing, you need to understand how Linux actually handles network traffic. The Linux kernel itself has a built-in packet filtering framework called Netfilter. Historically, System Administrators interacted with Netfilter using a command-line tool called iptables.

While incredibly powerful, iptables is notoriously complex. Writing a simple rule to block an IP address required memorizing a massive string of arcane syntax, chains, and target jumps. One misplaced character could instantly sever your connection to a remote server. Because of this steep learning curve, Canonical (the company behind Ubuntu) developed a frontend wrapper called UFW (Uncomplicated Firewall).

UFW is not a firewall itself; it is a translator. You type simple, plain-English commands into UFW, and UFW automatically writes the complex iptables (or modern nftables) rules under the hood. It brings enterprise-grade security down to a level anyone can understand.

2. Installing Uncomplicated Firewall

If you are running Ubuntu, Linux Mint, or Pop!_OS, UFW is likely already installed and waiting dormant on your system. If you are using Debian, Arch, or Fedora, you will need to pull it from your package manager.

$ sudo apt install ufw

# For Arch Linux users: sudo pacman -S ufw

# For Fedora users: sudo dnf install ufw

3. The Golden Rule: Deny Incoming, Allow Outgoing

Before we officially turn the firewall on, we must establish the baseline policies. The absolute standard security posture for any personal computer or web server is this: Nobody is allowed to initiate a connection with me unless I explicitly invited them, but I am allowed to reach out to the internet whenever I want.

Run these two commands to permanently set your default behavior:

$ sudo ufw default deny incoming

$ sudo ufw default allow outgoing

By defaulting to "deny," your machine effectively becomes invisible. If a malicious bot runs an automated port scan across your entire IP block looking for vulnerabilities, your computer will not reply to say "I'm closed" or "Connection Refused." It will simply drop the packets. The bot will assume the IP address is empty and move on.

4. Opening Specific Doors (TCP vs. UDP)

If your default policy is to deny everything, you will run into a problem if you actually want to host a service. For example, if you want to host a Minecraft server for your friends, or if you want to control your PC remotely using SSH, you need to manually poke holes in your new wall.

To open a door, you use the allow command. UFW is smart enough to recognize common application names, so you don't always have to memorize the port numbers.

$ sudo ufw allow ssh

# UFW automatically translates 'ssh' into Port 22.


$ sudo ufw allow 80/tcp

# Opens Port 80 specifically for standard HTTP web traffic.


$ sudo ufw allow 19132/udp

# Opens Port 19132 specifically for UDP traffic (often used for gaming servers).

Notice the /tcp and /udp tags? TCP ensures every packet arrives perfectly in order (essential for loading web pages and downloading files). UDP throws data as fast as possible without checking if it arrived (used for live video streaming and gaming where speed matters more than a dropped frame). By specifying the protocol, you tighten security by not opening both unnecessarily.

⚠️ The Remote Server Death Trap:
If you are configuring a remote VPS (like an AWS or DigitalOcean server) instead of the laptop sitting on your desk, you MUST run sudo ufw allow ssh BEFORE you turn the firewall on. If you throw the switch on a "deny all" firewall without explicitly opening the SSH port first, you will instantly sever your own connection and permanently lock yourself out of the remote server.

5. Advanced Rules: Rate Limiting & IP Blocking

UFW is capable of much more than just simple open/close policies. If you have Port 22 open for SSH, bots will constantly try to brute-force your password. You can use UFW to actively fight back.

Rate Limiting

Instead of using allow, you can use the limit command. This tells UFW: "Allow traffic on this port, but if a single IP address tries to connect more than 6 times in 30 seconds, automatically block them." This completely neutralizes brute-force dictionary attacks.

$ sudo ufw limit ssh

Explicit Denies & IP Blocking

If you check your server logs and see a specific IP address acting maliciously, you can permanently ban that exact IP address from communicating with your machine on any port.

$ sudo ufw deny from 203.0.113.50

# That specific IP address will never be able to reach your machine again.

6. Throwing the Switch & Logging

Your baseline rules are set. Your SSH port is explicitly allowed (and rate-limited). You are ready to bring the bouncer online.

$ sudo ufw enable

Firewall is active and enabled on system startup

To verify everything is working, and to see exactly which rules are currently active, use the verbose status command. It is good practice to run this immediately after enabling the firewall.

$ sudo ufw status verbose

Finally, if you want to keep track of every packet that gets dropped, you can turn on UFW logging. The firewall will silently write all blocked connection attempts to /var/log/ufw.log so you can audit them later.

$ sudo ufw logging on

Your perimeter is now secure. The fortress is locked. Next, we need to manage who is actually allowed to walk around inside the walls.
