SSH: The Encrypted Tunnel

Control your server from anywhere on Earth. Leave no trace. | 👤 By JR Nation

One of the greatest flexes in Linux is the ability to operate a computer sitting in a data center thousands of miles away as if you were sitting right in front of it. You do not need a graphical interface, a mouse, or a monitor. You just need a terminal and a protocol called SSH (Secure Shell).

Mastering SSH is an absolute requirement for managing web servers, Raspberry Pi clusters, or even just accessing your desktop from your laptop while traveling. But because it grants total command-line access to your machine over the internet, it is also the number one target for global cyberattacks. Let's learn how to build an impenetrable tunnel.

1. The Lore: The Death of Telnet

In the early days of the internet, system administrators controlled remote servers using a protocol called Telnet. There was a massive, fatal flaw with Telnet: it transmitted all data in "plaintext." If a hacker intercepted the traffic between your laptop and the server, they could literally read your administrator password as it floated across the network wires.

In 1995, a researcher named Tatu Ylönen got tired of password-sniffing attacks at his university. He invented SSH. SSH replaces plaintext with military-grade encryption. If a hacker intercepts an SSH connection today, all they see is a chaotic, mathematically impossible-to-decode stream of garbage data.

2. Asymmetric Cryptography (Public & Private Keys)

By default, SSH allows you to log into a remote machine using your standard user password. This is incredibly dangerous. If you expose SSH to the internet, botnets will try to brute-force guess your password thousands of times a minute.

The solution is to disable passwords entirely and use SSH Keys instead. This relies on Asymmetric Cryptography, which generates two mathematically linked files:

  • The Public Key: This is a padlock. You can give copies of this padlock to anyone. You put it on the remote server you want to control.
  • The Private Key: This is the key. It stays securely on your laptop and never travels across the internet. It is the only thing in the universe that can open your Public Key padlock.

3. Establishing the Connection

To connect to a remote machine, the remote machine must be running the SSH server program (the "daemon"), usually installed via sudo apt install openssh-server. Once it is running, you can connect from any other computer by opening a terminal and using this syntax:

$ ssh username@ip_address

# Example: ssh jaya@192.168.1.50

4. Generating and Distributing Keys

To ditch passwords, you first need to generate your cryptographic keypair on your local machine (your laptop). While old tutorials will tell you to generate "RSA" keys, the modern, ultra-secure, and lightning-fast standard is Ed25519.

$ ssh-keygen -t ed25519 -C "my_laptop"

# Press Enter to save it in the default hidden ~/.ssh folder.

# You can optionally add a passphrase to encrypt the file itself on your hard drive.

Now, you need to send your Public Key (the padlock) over to the remote server so it knows to trust you. Linux has a brilliant built-in command to do this automatically:

$ ssh-copy-id jaya@192.168.1.50

# It will ask for your password one last time to authenticate the copy.

5. Hardening the Daemon (Disabling Passwords)

Your keys are now in place. If you type ssh jaya@192.168.1.50, it will log you in instantly without asking for a password, because it cryptographically verified your Private Key in the background. But hackers can still try to guess your password! You must tell the remote server to reject all passwords universally.

To do this, you must edit the master SSH configuration file on the remote server:

$ sudo nano /etc/ssh/sshd_config

Scroll through the file and change these two critical settings:

  1. Change PasswordAuthentication yes to PasswordAuthentication no
  2. (Optional but recommended) Change the default Port 22 to a random high number like Port 49152. This won't stop a dedicated hacker, but it will hide your server from 99% of automated botnet scanners.

Save the file (Ctrl + O, Enter) and exit (Ctrl + X). Finally, restart the SSH engine to apply the new rules:

$ sudo systemctl restart sshd

⚠️ DO NOT LOSE YOUR PRIVATE KEY.
If you disabled password authentication and your laptop's hard drive dies (destroying your Private Key file), you are permanently locked out of the remote server. There is no "forgot password" button. Always keep a secure, encrypted backup of your ~/.ssh folder on an offline USB drive.
